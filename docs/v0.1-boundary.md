# v0.1 Scope and Boundary

This document defines the scope, non-goals, adoption boundary, review expectations,
non-canon posture, required packet pieces, and adjacent repo links for v0.1 of the
claim-evidence-ledger namespace.

## Status

Candidate namespace. Exploratory unless later adopted through a reviewed governance path.

## Scope

v0.1 defines the smallest useful, reviewable unit for a claim-evidence ledger:

- A single claim-evidence packet structure.
- The minimum fields required to express a claim, its evidence, source reference,
  verification status, and a cryptographic receipt reference.
- A JSON Schema describing that packet.
- One valid example, one valid fixture, and one invalid fixture.
- A receipt template for packets.
- Prior-art and adjacent-ecosystem references.
- This boundary document.

v0.1 does **not** define:

- A network protocol.
- A consensus mechanism.
- A storage backend.
- A governance process for adoption beyond this repository.
- A registry of authorities or issuers.

## Non-goals

- This repository is not a universal standard.
- This repository does not claim ownership of the broader field or terminology.
- This repository does not canonize HUMMBL/BaseN/Ownward concepts unless explicitly
  marked and audited.
- This repository must not include private, internal, or secret operational content.
- v0.1 does not attempt to be complete. It attempts to be small and reviewable.

## Adoption boundary

v0.1 artifacts are **candidate** material. Adoption is permitted only:

- Within this repository, for documentation and testing.
- By external parties experimentally, with the explicit understanding that the
  namespace is unstable and may change without notice.
- As reference material, not as a compliance or legal standard.

No v0.1 artifact is normative. No v0.1 artifact confers authority, endorsement, or
certification of any kind.

## Review expectations

v0.1 artifacts are expected to be reviewed for:

- Clarity and minimalism.
- Internal consistency between schema, example, and fixtures.
- Absence of private, internal, or secret operational content.
- Honest non-canon posture (no overclaims of novelty or ownership).
- Correctness of the invalid fixture (it must actually fail validation).

Reviewers are not expected to validate cryptographic properties of receipts in v0.1.
Receipts in v0.1 are structural placeholders for a future verification path.

## Non-canon posture

- The term "claim-evidence ledger" is descriptive, not proprietary.
- Prior art and adjacent ecosystem references are listed to acknowledge the field,
  not to claim derivation or endorsement.
- HUMMBL, BaseN, Ownward, and any other named concepts are mentioned only where
  relevant and are not canonized by this repository.
- Where a concept appears to overlap with existing public work, the existing public
  work takes precedence as reference.

## Required packet pieces

A v0.1 claim-evidence packet must contain, at minimum:

1. `schemaVersion` — the schema version this packet conforms to.
2. `packetStatus` — lifecycle status of the packet (e.g. `candidate`, `verified`).
3. `ledgerManifest` — identifying metadata for the ledger this packet belongs to:
   - `id`
   - `title`
   - `version`
   - `scope`
4. `authority` — who or what asserts this packet.
5. `claimEvidenceContract` — the core pair:
   - `claim` — the statement being made.
   - `evidenceType` — the kind of evidence offered.
   - `sourceReference` — where the evidence lives.
   - `verificationStatus` — current verification state.
   - `receiptRef` — reference to a cryptographic receipt.
6. `receiptRequirements` — what a valid receipt for this packet must provide.

See `schemas/claim-evidence-ledger-v0.1.json` for the normative shape.
